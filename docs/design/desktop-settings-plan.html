1. Why the menu item currently does nothing

The desktop app is a Tauri shell that always loads the bundled Nuxt SPA (never a remote workspace URL — see tauri.conf.json frontendDist: ../../web/.output/public). The native menu’s Preferences… item (Cmd/Ctrl+,) is wired as an event bridge:

menu.rs:208            "preferences" => { show_main_window(app); app.emit("menu://preferences", ()) }
apps/desktop/src/menu.ts:14   onMenuAction('preferences', cb)   // listen("menu://preferences")
useDesktopMenu.ts:20          onMenuAction('preferences', () => router.push('/dashboard/settings/desktop'))

The failure has two independent layers — both must be fixed:

  1. No listener is mounted outside the dashboard. useDesktopMenu() is called exactly once, from the dashboard layout (apps/web/app/layouts/dashboard.vue:36). When no workspace is connected, the user sits on /desktop/welcome, which uses layout: false — the composable never runs, so menu://preferences is emitted into the void. The click silently does nothing.
  2. The target route is unreachable without a workspace anyway. Even if navigation fired, /dashboard/settings/desktop is bounced by two guards:
    • app/middleware/desktop-workspace.global.ts:19-22 — desktop runtime + no active workspace → redirect to /desktop/welcome (allowed prefixes are only /desktop/welcome, /desktop/connect, /desktop/setup).
    • app/middleware/auth.ts:30-32 — the page carries middleware: 'auth'; unauthenticated desktop users are sent to /desktop/welcome.

Additionally, today’s “desktop settings” page (app/pages/dashboard/settings/desktop.vue) contains only a Launch at login toggle and static Default email app instructions, and it is not even linked from the settings hub (settings/index.vue) — it is reachable exclusively through the (broken-when-disconnected) menu item.

2. Target architecture

One always-available settings surface at /desktop/settings, living outside the /dashboard tree so it needs neither an active workspace nor authentication — the same trick /desktop/welcome already uses. The native menu item always opens this page, connected or not.

Key decisions and why

DecisionRationale
New route /desktop/settings instead of reusing /dashboard/settings/desktop The dashboard layout hard-requires an active workspace + auth. A standalone page (like welcome.vue) works in every state. The old route stays as a redirect so nothing that learned the old path breaks.
Menu listeners move from the dashboard layout into an app-level client plugin Plugins run on every route and in every window of the SPA, so the menu works from welcome, connect, setup, dashboard — everywhere. This is the actual bug fix.
Keep the Rust side event-based (menu://preferences), don’t hardcode navigation in menu.rs The JS side owns routing state and can evolve the destination without a Rust release. handle_menu_event already calls show_main_window first, which is exactly right.
New store file settings.json rather than growing workspaces.json workspaces.json is identity/connection metadata consumed by the boot path; mixing mutable preferences in risks churn on a boot-critical file. store:default is already granted in capabilities/default.json — no capability/permission changes needed.
Two tiers of workspace settings: device-local (in settings.json, keyed by workspace id) and server-side (existing Convex-backed /dashboard/settings hub, linked out) Device-local per-workspace prefs (e.g. mute) must work even when that workspace isn’t the active one or is unreachable. Server settings already have a mature home; duplicating them offline would be wrong.

3. Settings model & storage

Each setting has exactly one owner. Scope legend: GLOBAL device-wide, spans all workspaces · WORKSPACE per-workspace · OS-OWNED lives in the operating system, surfaced but not stored by us.

Store schema (settings.json, key state)

{
  "version": 1,
  "global": {
    "autoCheckUpdates": true,
    "notificationsEnabled": true,
    "showUnreadBadge": true,
    "startupWorkspaceId": null
  },
  "workspaces": {
    "<workspaceId>": { "muteNotifications": false }
  }
}

Missing file, unknown keys, or older version → merge over typed defaults (same defensive pattern as loadWorkspaceStore()). On the plain web build (no Tauri) the composable falls back to localStorage key owlat:desktop-settings so the page remains developable in a browser.

4. Implementation phases

Phase 1 — Fix the dead menu item (menu works from any state)

Smallest shippable slice: clicking Settings always opens a page, even with zero workspaces. ~½ day.

  1. NEW apps/web/app/plugins/1.desktop-menu.client.ts — app-level registration of all three menu bridges (replaces the layout-scoped composable). Ordering: prefix 1. so it runs after the 0.desktop-workspace boot plugin. Sketch: 
    export default defineNuxtPlugin(async () => {
  if (!isDesktopRuntime()) return;
  const { getCurrentWebviewWindow } = await import('@tauri-apps/api/webviewWindow');
  if (getCurrentWebviewWindow().label !== 'main') return; // app.emit broadcasts to
                                                          // ALL windows — the compose
                                                          // window must not also react
  const router = useRouter();
  const { onMenuAction } = await import('@owlat/desktop/src/menu');
  await onMenuAction('preferences',   () => router.push('/desktop/settings'));
  await onMenuAction('new-workspace', () => router.push('/desktop/welcome'));
  await onMenuAction('check-updates', () =>
    window.dispatchEvent(new Event('owlat:check-updates')));
});
    No teardown needed — plugin listeners live for the lifetime of the webview.
  2. apps/web/app/layouts/dashboard.vue:36 — remove the useDesktopMenu(); call. app/composables/useDesktopMenu.ts — delete (its one consumer is gone; keeping it invites double registration).
  3. apps/web/app/middleware/desktop-workspace.global.ts:12 — add '/desktop/settings' to the allowed-prefix list so the global guard stops redirecting it to /desktop/welcome.
  4. NEW apps/web/app/pages/desktop/settings.vue — page shell in the style of welcome.vue: definePageMeta({ layout: false }), no auth middleware, renders its own <DesktopTitlebar />, non-desktop fallback message (same pattern as desktop.vue:34-39). Left nav with two groups — Global and Workspaces — content sections filled in Phases 2–3. For Phase 1 it can ship with just the moved Launch at login + Default email app content so the menu item lands somewhere real immediately.
  5. apps/web/app/pages/dashboard/settings/desktop.vue — replace the body with a redirect (navigateTo('/desktop/settings', { replace: true })) so the old deep-link and any muscle memory keep working.

Phase 2 — Global settings store & page sections

Real persisted global settings. ~1 day.

  1. NEW apps/desktop/src/settings.ts — store bridge, mirroring workspace.ts exactly: FILE = 'settings.json', KEY = 'state', loadSettingsStore() / saveSettingsStore(), warn-and-default on failure. Plain JSON in/out to stay decoupled from web types.
  2. NEW apps/web/app/lib/desktop/settingsTypes.ts — DesktopSettings / GlobalSettings / WorkspaceLocalSettings types, SETTINGS_VERSION, defaultDesktopSettings(), and normalizeSettings(raw) (defaults-merge + future migrations). Mirrors how workspaceTypes.ts pairs with workspace.ts.
  3. NEW apps/web/app/composables/useDesktopAppSettings.ts — module-level reactive singleton (the useDesktopWorkspaces pattern): hydrate once on first use, expose settings (reactive), isReady, and typed mutators (setGlobal(key, value), setWorkspaceLocal(id, key, value)). Persistence: debounced (~300 ms), single-flight write via the bridge; localStorage fallback off-desktop. Name deliberately avoids colliding with the existing useDesktopSettings.ts (autostart), which stays as-is.
  4. apps/web/app/pages/desktop/settings.vue + NEW apps/web/app/components/desktop/settings/*.vue — global sections:
    • Appearance — theme picker bound to useAppTheme() (reuse the control already living in dashboard/settings/index.vue:585-631; extract it into a shared component rather than duplicating).
    • Startup — Launch at login (existing useDesktopSettings/autostart bridge) + Open at startup workspace picker (last active or a pinned workspace from useDesktopWorkspaces().workspaces).
    • Updates — current version (@tauri-apps/api/app getVersion()), auto-check toggle, Check now button dispatching owlat:check-updates.
    • Notifications — enable toggle + unread-badge toggle.
    • Default email app — informational panel moved verbatim from the old desktop.vue.

Phase 3 — Workspace-specific settings

Per-workspace section that degrades gracefully when disconnected. ~1 day.

  1. settings.vue Workspaces section — one card per entry in useDesktopWorkspaces().workspaces (this list is available without auth — it’s local metadata):
    • Label + host, active indicator (activeId).
    • Accent color — reuse existing setWorkspaceAccent().
    • Mute notifications — writes workspaces[id].muteNotifications in settings.json.
    • Sign out / Remove from this device — reuse existing signOutWorkspace() / removeWorkspace() (destructive: confirm dialog).
    • “Open workspace settings” → server-side settings. If the workspace is active: router.push('/dashboard/settings'). If not: switchTo(id) — note this reloads the webview by design (useDesktopWorkspaces.ts:294-330); pass the pending destination through the existing workspace-switch sessionStorage channel (app/lib/desktop/workspaceSwitch.ts) so the reload lands on /dashboard/settings instead of the inbox.
  2. Empty state (zero workspaces) — explains that workspace settings appear once connected; CTA button → /desktop/welcome. This is the exact “menu pressed while not connected” scenario from the bug report: the user still gets a working page with all global settings plus a clear path to connect.
  3. useDesktopWorkspaces.ts removeWorkspace() — prune the removed id from settings.json → workspaces so orphaned prefs don’t accumulate.
  4. apps/web/app/pages/dashboard/settings/index.vue (settingsSections, :289-390) — add a “Desktop app” card (gated on isDesktop) linking to /desktop/settings, so the surface is also discoverable from inside a workspace, not only via the native menu.

Phase 4 — Wire settings into behavior, native polish

Settings actually change behavior; native labels follow platform conventions. ~½–1 day.

  1. apps/web/app/lib/desktop/updater.client.ts — setupUpdateChecks() skips the boot-time runUpdateCheck() when global.autoCheckUpdates === false. The explicit owlat:check-updates listener stays unconditional (a manual check must always work).
  2. apps/web/app/composables/useDesktopNotifications.ts — gate delivery on global.notificationsEnabled and on the active workspace’s muteNotifications; gate updateUnreadBadge() on global.showUnreadBadge (emit 0/clear when disabled so a stale badge doesn’t linger).
  3. apps/web/app/plugins/0.desktop-workspace.client.ts — during boot, if global.startupWorkspaceId is set and still exists in the store, seed it as the active workspace instead of the persisted last-active id.
  4. apps/desktop/src-tauri/src/menu.rs:58-64 — rename the item: macOS 13+ convention is “Settings…” (was “Preferences…”), same on Windows/Linux. Keep the menu id "preferences" and the menu://preferences event name — they’re internal contract, renaming them buys nothing and risks a missed call site.
  5. Update the module docs that describe the old flow (menu.rs header comment, apps/desktop/src/menu.ts header, boot-plugin comments) so they name the plugin instead of useDesktopMenu.

5. Edge cases & risks

CaseHandling
Compose window double-handling. app.emit broadcasts menu://… to every webview window; the compose window runs the same SPA and the same plugins.The new plugin returns early unless getCurrentWebviewWindow().label === 'main' (step 1.1). Without this guard the compose window would also navigate — visibly wrong.
Menu clicked before the SPA finishes booting. Rust emits immediately; listeners register during plugin init.Accept the tiny window (sub-second) — Tauri events are not queued, and show_main_window already focuses the app. Not worth a replay mechanism.
Corrupt / missing / future-versioned settings.json.normalizeSettings() always merges over defaults and never throws; unknown keys preserved on write-back where cheap, else dropped. Same resilience contract as loadWorkspaceStore().
Two windows writing settings concurrently.Only the main window renders the settings page; the compose window is read-only at boot. Debounced single-flight saves in the composable prevent self-races. Acceptable without store-change events for now.
Plain web build (no Tauri)./desktop/settings shows the same “only relevant in the desktop app” fallback the old page had; the plugin no-ops (isDesktopRuntime() guard). localStorage fallback keeps the page testable in browser dev.
startupWorkspaceId points at a removed workspace.Boot falls back to last-active; removeWorkspace() also nulls it when it matches (Phase 3.3).
“Open workspace settings” for an inactive workspace reloads the webview.Inherent to the existing switchTo() design (singleton re-seeding). Route the post-reload destination through the existing workspaceSwitch.ts sessionStorage channel; show the existing switch skeleton.
Auth-gated server settings.Never fetched on /desktop/settings itself — the page only links into /dashboard/settings, where the existing auth middleware does its job. No new auth surface.

6. Testing & verification

All tests via vitest (never bun test — it skips the Nuxt auto-import polyfill setup file).

  • Unit — apps/desktop: src/__tests__/settings.test.ts mirroring the existing bridge tests: load returns defaults on missing/corrupt store, round-trips state, warns without throwing on store failure.
  • Unit — apps/web: normalizeSettings() (defaults merge, version handling); useDesktopAppSettings mutators + debounce (mock bridge); middleware allowlist — /desktop/settings passes the disconnected-desktop guard while /dashboard/settings still redirects.
  • Manual matrix (cd apps/desktop && bun tauri dev), driven per platform:
    1. Fresh profile, zero workspaces, on /desktop/welcome → menu Settings / Cmd+, → lands on /desktop/settings, global sections usable, Workspaces shows empty state + connect CTA. (The reported bug, verified fixed.)
    2. Connected, inside dashboard → same entry points work; “Desktop app” card visible in the settings hub; old /dashboard/settings/desktop URL redirects.
    3. Compose window open → menu Settings navigates the main window only.
    4. Toggle auto-update off → relaunch → no boot toast; Check now still works. Toggle notifications/badge/mute → observe delivery + badge. Restart after changing startup workspace.
    5. Delete settings.json from the app data dir → app boots clean on defaults.
  • Repo gates: bun run ci:test, bun run lint before the PR.

7. Open decisions

  1. Menu label: plan says rename “Preferences…” → “Settings…” to match the user-facing wording (“settings menu”) and macOS 13+ / Windows convention. Trivial to keep the old label if branding prefers it.
  2. Server-side per-user notification scope (notifyAbout via api.mail.settings): currently stays in the workspace hub. It could later be surfaced read-only on the desktop page for the active workspace, but that adds an auth dependency to an intentionally auth-free page — deferred.
  3. Settings sync across devices: out of scope — settings.json is deliberately device-local; anything that should roam belongs in Convex per-user settings instead.
  4. Store-change events between windows (live sync to the compose window): skipped for now; revisit if a compose-relevant setting (e.g. spellcheck) is added later.
