Saved blocks let you build reusable content sections — a header, a footer, a call-to-action — once and insert them into any email. Unlike a copy-paste snippet, a saved block stays **linked**: when you edit the block, every email that uses it updates to match. Change your footer in one place and every template and transactional email carrying it re-renders with the new version.

## Why use saved blocks

- **Consistency** — your brand header and footer look identical across every email.
- **Speed** — drop in a polished CTA section in seconds instead of rebuilding it.
- **One-edit maintenance** — because inserted blocks are linked, fixing a typo or swapping a logo in a saved block flows out to every email that uses it. No hunting through old templates.

## How linked blocks work

When you insert a saved block into an email, the editor doesn't paste a static copy — it inserts a **linked instance**. Each instance remembers which saved block it came from, and the editor marks it with a **Linked block** label (carrying the block's name) above the content.

What that means in practice:

| When you… | What happens |
| --- | --- |
| Edit a saved block's **content** | Every template and transactional email that uses it is updated and re-rendered to match. |
| Rename a saved block | The label shown on every linked instance updates to the new name. |
| Delete a saved block | It is detached from every email that uses it — those emails keep the content as a normal, now-independent block. |
| Detach a linked instance | That one email gets an independent copy and stops receiving updates from the library. |

::callout{type="warning" title="Edits cascade"}
Editing a saved block's content changes every email that already uses it. This is the point of saved blocks — but it means a careless edit can alter live templates. If you want a one-off variation, **detach** the instance first (see below) so your change stays local.
::

The re-render of consuming emails happens in the background. After you save a content change, the affected templates and transactional emails are marked stale and re-rendered shortly after by a retrying render pool — so it may take a moment for an email's rendered HTML to catch up.

## Saving a block

Saved blocks are created and edited from the **Mail > Blocks** page.

::steps

### Open the Blocks page

Go to **Mail > Blocks** and click **New Block**.

### Name it

Give the block a descriptive **Name** (e.g. "Standard Email Header" or "Product CTA with Image") and an optional **Description**. The description helps you and your teammates pick the right block later.

### Build the content

Click **Create Block** — this creates the block and opens a dedicated block editor where you add and arrange its content. The saved block can be a single block or a group of several blocks. Your changes are the block's content.

::

A saved block can hold one block or many. Multi-block groups show a small **N blocks** badge on the Blocks page so you can tell them apart at a glance.

## Inserting a saved block

Insert saved blocks from inside the [Email Editor](/guide/email-editor) using the slash menu.

::steps

### Open the slash menu

Type `/` where you want the block to go.

### Find your block

Scroll to the **Saved** group at the bottom of the menu, or start typing part of the block's name to filter the list.

### Insert it

Select the block. It's inserted as a linked instance right after the block you're on (or at the end of the document), marked with its **Linked block** label.

::

::callout{type="info" title="Detaching a linked instance"}
To turn a linked instance into an independent copy, click the **Detach** icon on its **Linked block** label and confirm. The detached block keeps its current content but stops receiving updates from the saved block library.
::

## Managing saved blocks

Open **Mail > Blocks** to see every saved block. You can search by name or description, and sort by **Most Recent**, **Most Used**, or **Name (A-Z)**. Each block shows its **usage count** (how many emails currently link to it) and, for multi-block groups, a **N blocks** badge.

Per block, you can:

- **Edit Content** — open the block editor to change what's inside (this is the edit that cascades to every linked email).
- **Settings** — rename the block or update its description.
- **Duplicate** — create an independent copy (named "… (Copy)") to use as a starting point.
- **Delete** — remove the block from the library.

::callout{type="warning" title="Deleting a block in use"}
Deleting a saved block that's used by one or more emails will warn you first. On delete, the block is **detached** from every email that uses it: those emails keep the content as an independent block, but it's no longer linked to anything.
::

## Best practices

- **Save your header and footer early** — these are the blocks you'll reuse most, so set them up before building your first campaign. Edit them once and every email updates.
- **Keep blocks focused** — a "Header" block should just be the header, not the header plus the first content section. Smaller blocks are more versatile.
- **Name blocks descriptively** — "Holiday Sale CTA (Red)" is better than "CTA 2". The name is also what shows on every linked instance in the editor.
- **Detach before customizing** — if one email needs a tweaked version of a shared block, detach that instance first so your change doesn't cascade to everyone else.
- **Review periodically** — use the **Most Used** sort and the usage count to spot unused blocks, then delete them to keep the library clean.
